# 五层职责说明（App / Service / Platform / Impl / Vendor）

> 本项目：26 电赛 · 平台化循迹小车。架构沿用课程"系统第一刀分层"思想。
> 分层不是为了文件夹好看，而是**为了让后续变化有地方被拦住**——换主控、换灰度/摄像头、换电机、改循迹算法、加通信协议，都不该扩散到整个工程。

**一句话立规矩**：
> App 不直接碰 HAL · Service 不直接碰 Vendor · Platform 只定义接口 · Impl 才做具体适配 · Vendor 只保留厂家底座

---

## 五层职责表

| 层 | 职责定位 | 该放什么（循迹小车预期） | 禁区 | 一句话原则 | 工程落点 |
|---|---|---|---|---|---|
| **App** | 产品业务流程 | 上电流程 / 循迹主流程 / 状态机（出发-循迹-路口-停车）/ 调参入口 | 直接调 HAL、直接操作 GPIO/ADC/PWM/I2C | App 只描述**要干什么**，不关心硬件怎么干 | `01_App/` |
| **Service** | 可复用、带策略的系统能力 | service_tracking（循迹感知+决策）/ service_motor（速度闭环）/ service_comm / service_log ... | 直接碰 Vendor、绕过 Platform 调 Impl/HAL | Service 把底层能力**组织成可复用能力** | `02_Service/service_xxx/` |
| **Platform** | 平台抽象层：定义统一接口/对象协议，**不做实现** | 统一接口 / 统一错误码 / 统一数据结构 / ops 函数指针 / ctx 上下文 | 芯片头文件（如 `stm32xxxx_hal.h`）、`HAL_xxx()`、`MX_xxx_Init()` | Platform 只**定义能力**，不绑定具体芯片 | `03_Platform/`（common/mcu/os/bsp/middleware） |
| **Impl** | 把 Platform 抽象接口落到具体平台 | 主控 IO/PWM/ADC/UART 实现、OS 适配、器件级 BSP（灰度、编码器、电机驱动板）、板级资源映射 | 反向调用 Service / App | Impl 是 Platform 与 Vendor 之间的**适配层** | `04_Impl/`（board/mcu/os/bsp/middleware） |
| **Vendor** | 厂家与第三方底座 | HAL / CMSIS / CubeMX 生成码 / RTOS / 第三方算法库 | 反向调用上层、直接改源码 | Vendor 只提供**底座**，不理解产品业务 | `05_Vendor/`（+ 物理保留 Core/Drivers/Middlewares） |

> 调用链示例（一条信号穿过五层）：
> `app_main()` → `service_motor_set_speed()` → `platform_pwm_set_duty()` → `impl_xxx_pwm_set()` → `HAL_TIM_PWM_xxx()`

## 边界判断（课程 Q&A 提炼）

| 问题 | 结论 |
|---|---|
| BSP 放哪层？ | 板级资源映射（哪个 ADC 通道是灰度、哪个 TIM 是电机 PWM）→ `Impl/impl_board`；器件级驱动（灰度、编码器、电机驱动）→ `Impl/impl_bsp` |
| CubeMX 的 Core 目录算哪层？ | 逻辑上属 Vendor/生成码；**物理上别移动**（CubeMX 会重新生成），文档说明即可 |
| `main.c` 属于哪层？ | 启动胶水代码，**只做启动转发** `app_main()`，不写业务 |
| Platform 能 include 芯片 HAL 吗？ | 不能。一旦 include 就等于 Platform 绑死该芯片 |
| Impl 里很多函数是空的正常吗？ | 正常。第一刀先**留位置**，后面再长血肉 |
