# 依赖方向说明（Dependency Direction）

> 本项目：26 电赛 · 平台化循迹小车。配套见同目录《五层职责说明》。

**核心规则**：依赖只能**自上而下**，下层永远不知道上层存在。

```
App  →  Service  →  Platform  →  Impl  →  Vendor
（单向向下，不可反向、不可跨层向下穿透 Platform）
```

---

## 一、依赖规则表（允许 / 禁止）

| 主体层 | 允许依赖 | 禁止依赖 | 说明 |
|---|---|---|---|
| **App** | Service | 直接 include HAL / 直接调 Impl / 直接调 Vendor | App 只通过 Service 拿能力 |
| **Service** | Platform | 直接调 Vendor / include HAL / 绕过 Platform 调 Impl | Service 面向 Platform 接口编程 |
| **Platform** | platform_common（同层公共类型/错误码） | 绑定具体芯片、依赖 Impl 具体实现 | Platform 只定义接口，由 Impl 反向实现 |
| **Impl** | Platform（实现其接口）+ Vendor | **反向调用 Service / App** | Impl 是适配层，下不知上 |
| **Vendor** | 无（最底座） | 反向调用任何上层 | Vendor 不理解产品业务 |

> 关键约束：**Platform 定标准，Impl 做适配**。Platform 里写"我需要一个 PWM 占空比接口 `platform_pwm_set_duty()`"，Impl 里才写"这个接口在主控上怎么调 `HAL_TIM_xxx()`"。

---

## 二、标准调用链模板

```
<App 业务函数>()
    → <Service 能力函数>()
        → <Platform 抽象接口>()
            → <Impl 适配函数>()
                → <Vendor/HAL 函数>()
```

### 本项目参考（电机 PWM）

```
app_main()                              [01_App]
    → service_motor_set_speed(v)         [02_Service]   速度闭环策略
        → platform_pwm_set_duty(ch, d)    [03_Platform]  抽象接口
            → impl_xxx_pwm_set(ch, d)      [04_Impl]      主控适配
                → HAL_TIM_PWM_xxx()         [05_Vendor]    厂家底座
```

---

## 三、include 禁区检查表（落地校验）

| 检查项 | 期望 |
|---|---|
| `01_App/**` 是否 include HAL？ | ❌ 不允许 |
| `02_Service/**` 是否 include HAL？ | ❌ 不允许 |
| `03_Platform/**` 是否 include 芯片 HAL？ | ❌ 不允许（绑死芯片） |
| `04_Impl/**` 才能看到 HAL？ | ✅ 只有 Impl 能 |
| Vendor 是否反向 include 上层？ | ❌ 不允许 |
| Impl 是否反向调用 Service？ | ❌ 不允许 |

---

## 四、本课验收三连

- [ ] **工程能编译通过**
- [ ] **业务入口已从 `main.c` 转到 `app_main()`**（main.c 只做启动转发）
- [ ] **依赖没有乱**：App/Service 不 include HAL、Platform 不绑芯片、只有 Impl 见 HAL、Vendor 不知上层存在
